# OPENAPI CODEGEN IS OBSOLETE

**A Technical Assessment**

*Prepared by the Straylight Framework Team*

---

## Abstract

This document presents a technical argument that mainstream OpenAPI code
generators—openapi-generator, oapi-codegen, swagger-codegen, and their
derivatives—represent an obsolete approach to API client generation. We
demonstrate this through systematic comparison with Zeitschrift, a scope
graph-based system that treats API specifications as a binding problem
rather than a template expansion problem. The argument is not that existing
tools lack utility, but that their architectural foundations—mustache
templates, loose typing, and runtime validation—have been superseded by
approaches offering stronger guarantees with superior maintainability.

---

## 1. The Template Problem

Every mainstream OpenAPI generator is built on string templates:

| Tool | Template Engine | Templates |
|------|-----------------|-----------|
| openapi-generator | Mustache | 3,000+ files |
| swagger-codegen | Mustache | 2,500+ files |
| oapi-codegen | Go text/template | ~20 files |
| openapi-typescript | Handlebars | ~15 files |

This architecture has fundamental problems:

### 1.1 Templates Are Untyped

```mustache
{{#operations}}
{{#operation}}
public {{#returnType}}{{returnType}}{{/returnType}}{{^returnType}}void{{/returnType}} 
    {{operationId}}({{#allParams}}{{dataType}} {{paramName}}{{^-last}}, {{/-last}}{{/allParams}})
{{/operation}}
{{/operations}}
```

This template:
- Has no type checking
- Fails silently on missing variables
- Mixes logic with presentation
- Cannot be refactored safely

### 1.2 Template Proliferation

openapi-generator supports 50+ languages. Each language has 50-200 template
files. The result:

```
openapi-generator/
  modules/openapi-generator/src/main/resources/
    Java/              87 templates
    kotlin/            62 templates  
    python/            54 templates
    typescript-fetch/  23 templates
    rust/              41 templates
    ...
```

**Total: 3,000+ templates to maintain.**

When the OpenAPI spec adds a feature (e.g., `oneOf` discriminators), every
template set must be updated independently. They diverge. Bugs accumulate.

### The Zeitschrift Alternative

Zeitschrift has **one IR** (intermediate representation) and **one codegen
module per language**:

```haskell
-- The entire interface
class Codegen (l :: Lang) where
    generateClient :: SLang l -> ClientConfig -> API -> Either CodegenError [GeneratedFile]
    generateServer :: SLang l -> ServerConfig -> API -> Either CodegenError [GeneratedFile]
    generateTypes  :: SLang l -> API -> Either CodegenError [GeneratedFile]
    fileExtension  :: SLang l -> Text
```

Each language implements this interface with **typed AST construction**, not
string interpolation:

```haskell
-- TypeScript client generation (excerpt)
clientModule :: ClientConfig -> API -> Doc ann
clientModule ClientConfig{..} API{..} =
    vsep
        [ "export class " <> pretty (tsTypeName apiTitle) <> "Client {"
        , "  private config: Required<Pick<ClientConfig, 'baseUrl' | 'timeout'>>;"
        , "  private fetchFn: typeof fetch;"
        , ...
        ]
```

The `Doc` type is a **structured document**, not a string. It pretty-prints
correctly, handles indentation automatically, and can be analyzed/transformed
before rendering.

## 2. The Type System Gap

### 2.1 Generated Code Is Stringly-Typed

openapi-generator produces code like this:

```java
// Generated by openapi-generator
public class PetApi {
    public Pet getPetById(Long petId) throws ApiException {
        Object localVarPostBody = null;
        
        // verify the required parameter 'petId' is set
        if (petId == null) {
            throw new ApiException(400, "Missing required parameter 'petId'");
        }
        
        String localVarPath = "/pet/{petId}"
            .replaceAll("\\{" + "petId" + "\\}", apiClient.escapeString(petId.toString()));
        
        // ... string manipulation continues
    }
}
```

Problems:
- Path construction via string replacement
- Runtime null checks instead of compile-time guarantees
- `Object localVarPostBody` loses all type information

### 2.2 Zeitschrift: Type-Safe by Construction

```rust
// Generated by zeitschrift
impl PetsClient {
    pub async fn get_pet_by_id(&self, pet_id: i64) -> Result<Pet, Error> {
        let path = format!("/pet/{}", pet_id);
        self.request(Method::GET, &path)
            .send()
            .await?
            .json()
            .await
    }
}
```

The Rust type system guarantees:
- `pet_id` is `i64`, not `Option<i64>`
- Return type is `Result<Pet, Error>`, not `Object`
- No runtime path manipulation errors

### 2.3 TypeScript: Zod Schema Inference

openapi-typescript-codegen generates:

```typescript
// Generated by openapi-typescript-codegen
export type Pet = {
    id?: number;
    name?: string;
    status?: 'available' | 'pending' | 'sold';
};

// No runtime validation!
```

Zeitschrift generates:

```typescript
// Generated by zeitschrift
import { z } from 'zod';

export const PetSchema = z.object({
    id: z.number(),
    name: z.string(),
    status: z.enum(['available', 'pending', 'sold']),
});

export type Pet = z.infer<typeof PetSchema>;

// Client validates at runtime
const pet = PetSchema.parse(await response.json());
```

The schema **is** the type. Validation happens automatically. Mismatches are
caught, not silently accepted.

## 3. The Scope Graph Revolution

### 3.1 The Problem: Dangling References

OpenAPI specs reference schemas:

```yaml
paths:
  /pets/{petId}:
    get:
      responses:
        '200':
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Pet'

components:
  schemas:
    Pet:
      type: object
      properties:
        category:
          $ref: '#/components/schemas/Category'  # Does this exist?
```

Traditional generators resolve references at generation time. If a reference
dangles, you get:
- A runtime error (bad)
- A generated type called `AnyType` (worse)
- Silent success with broken code (worst)

### 3.2 Zeitschrift: References Must Resolve

Zeitschrift builds a **scope graph** before generating any code:

```
┌─────────────────────────────────────────────────────────────────┐
│                       Scope Graph                               │
│  The binding structure of your entire API                       │
│  "If this builds, every reference resolves"                     │
└─────────────────────────────────────────────────────────────────┘
```

Every `$ref` is a name resolution query in the graph. If it doesn't resolve,
the build fails **before** any code is generated.

```bash
$ zeitschrift check openapi.yaml
error: unresolved reference '#/components/schemas/Category'
  --> openapi.yaml:15:23
   |
15 |           $ref: '#/components/schemas/Category'
   |                 ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ not found in scope
   |
   = help: did you mean '#/components/schemas/CategoryResponse'?
```

This is Eelco Visser's scope graph theory applied to API specifications.

### 3.3 Cross-Artifact References

The scope graph spans **all artifacts**, not just the OpenAPI spec:

```
┌─────────────────────────────────────────────────────────────────┐
│                        Lean 4 Proofs                            │
│  "If this compiles, the claim is proven"                        │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                       Scope Graph                               │
│  Code, docs, specs, proofs—all cross-referenced                 │
└─────────────────────────────────────────────────────────────────┘
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
        ┌──────────┐   ┌──────────┐   ┌──────────┐
        │  Human   │   │  Agent   │   │  Prover  │
        │  Docs    │   │  Tokens  │   │  Terms   │
        └──────────┘   └──────────┘   └──────────┘
```

Documentation references the API. The API references types. Types reference
constraints. **All links are validated.**

## 5. The Language Quality Gap

### 5.1 openapi-generator Rust Output

```rust
// Generated by openapi-generator
#[derive(Clone, Debug, PartialEq, Default, Serialize, Deserialize)]
pub struct Pet {
    #[serde(rename = "id", skip_serializing_if = "Option::is_none")]
    pub id: Option<i64>,
    #[serde(rename = "name")]
    pub name: String,
    #[serde(rename = "status", skip_serializing_if = "Option::is_none")]
    pub status: Option<String>,  // Should be an enum!
}
```

Problems:
- `status` is `Option<String>` instead of an enum
- Verbose serde attributes on every field
- `Default` derived but fields have required constraints

### 5.2 Zeitschrift Rust Output

```rust
// Generated by zeitschrift
use serde::{Deserialize, Serialize};

#[derive(Clone, Debug, PartialEq, Serialize, Deserialize)]
pub struct Pet {
    pub id: i64,
    pub name: String,
    pub status: PetStatus,
}

#[derive(Clone, Copy, Debug, PartialEq, Eq, Hash, Serialize, Deserialize)]
#[serde(rename_all = "lowercase")]
pub enum PetStatus {
    Available,
    Pending,
    Sold,
}
```

- Proper enum extraction
- No unnecessary `Option` wrappers
- Idiomatic Rust (no redundant serde renames)
- `Copy` on small enums

### 5.3 TypeScript Quality

openapi-typescript:
```typescript
export interface Pet {
  id?: number;
  name?: string;  // Required in spec, optional in output!
  status?: "available" | "pending" | "sold";
}
```

Zeitschrift:
```typescript
export const PetSchema = z.object({
  id: z.number(),
  name: z.string(),
  status: z.enum(["available", "pending", "sold"]),
});

export type Pet = z.infer<typeof PetSchema>;
```

Required fields are required. The type matches the spec.

## 6. The Maintenance Burden

### 6.1 Template Maintenance

openapi-generator requires:
- Java development environment
- Maven build
- Understanding of Mustache templating
- Knowledge of internal data models

To fix a bug in TypeScript generation:
1. Find the template (which of 23?)
2. Understand the data model
3. Modify the template
4. Rebuild the entire generator
5. Test against multiple specs
6. Submit PR, wait for review

### 6.2 Zeitschrift Maintenance

Zeitschrift is a Haskell library. To modify TypeScript generation:

```haskell
-- Zeitschrift/Codegen/TypeScript.hs
generateClientMethod :: Endpoint -> Doc ann
generateClientMethod Endpoint{..} =
    vsep
        [ "async " <> pretty (tsMethodName endpointOperationId) 
            <> parens (parameters endpointParameters)
            <> ": Promise<" <> responseType endpointResponses <> "> {"
        , indent 2 body
        , "}"
        ]
```

- Typed AST manipulation
- Compiler catches errors
- Refactoring is safe
- Changes are localized

## 7. The Runtime Story

### 7.1 Traditional: Generate and Pray

```
OpenAPI Spec → Generator → Source Code → Compile → Ship

                              ↓
                    "Hope the types match"
```

No runtime validation. The generated client trusts the server completely. If
the server returns malformed data, you get:
- Deserialization errors (if lucky)
- Null pointer exceptions (if unlucky)
- Silent data corruption (if cursed)

### 7.2 Zeitschrift: Validate at Boundaries

```
OpenAPI Spec → Zeitschrift → Source Code + Schemas → Compile → Ship

                                        ↓
                           Runtime validation at boundaries
```

TypeScript:
```typescript
const response = await this.fetchFn(url);
const data = await response.json();
return PetSchema.parse(data);  // Validated!
```

Rust:
```rust
let response = self.client.get(&url).send().await?;
let pet: Pet = response.json().await?;  // serde validates
```

Python:
```python
response = await self.client.get(url)
return Pet.model_validate(response.json())  # Pydantic validates
```

**Trust boundaries are explicit.** External data is validated before entering
your type system.

## 9. The Multi-Spec Problem

### 9.1 Traditional: One Spec at a Time

```bash
openapi-generator generate -i pet-api.yaml -g typescript ...
openapi-generator generate -i user-api.yaml -g typescript ...
# Now manually merge the outputs
```

Multiple specs require manual coordination:
- Shared types must be extracted
- Naming conflicts must be resolved
- Import paths must be configured

### 9.2 Zeitschrift: Unified Graph

```yaml
# zeitschrift.yaml
sources:
  - pet-api.yaml
  - user-api.yaml
  - shared-types.yaml

output:
  typescript:
    path: ./src/generated
    client: true
```

Zeitschrift builds **one scope graph** from all sources. Shared types are
automatically deduplicated. Conflicts are reported as errors:

```
error: conflicting definitions for 'Pet'
  --> pet-api.yaml:42:3
   |
42 |   Pet:
   |   ^^^ first definition
   |
  --> user-api.yaml:89:3
   |
89 |   Pet:
   |   ^^^ conflicts with previous definition
   |
   = help: rename one definition or extract to shared-types.yaml
```

## 10. The Agent Dimension

### 10.1 The Reader Might Be an Agent

Traditional documentation is prose optimized for humans:

> "The Pet API allows you to manage pets in the store. To get a pet by ID,
> call the `getPetById` method with the pet's numeric identifier..."

An AI agent reading this might:
- Hallucinate parameter names
- Invent endpoints that don't exist
- Confuse similar-sounding operations

### 10.2 Scope Graphs Are Agent-Native

The scope graph is a **formal structure** that agents can traverse:

```json
{
  "scope": "PetApi",
  "declarations": [
    {
      "name": "getPetById",
      "kind": "operation",
      "parameters": [
        { "name": "petId", "type": "integer", "required": true }
      ],
      "returns": { "$ref": "#/Pet" }
    }
  ],
  "references": [
    { "from": "getPetById.returns", "to": "Pet", "resolved": true }
  ]
}
```

No hallucination possible. References either resolve or they don't. The
structure is the documentation.

## 12. Conclusion

OpenAPI codegen tools are not obsolete in the market. They are obsolete **in
principle**.

The evidence:

1. **Template architecture** is untyped, unmaintainable, and divergent
2. **Generated types** are often wrong (optionality, enums, constraints)
3. **No scope analysis** means dangling references go undetected
4. **Configuration explosion** requires expertise in each generator
5. **Language quality** varies wildly between templates
6. **No runtime validation** in most generated clients
7. **Documentation** is copy-paste, not linked
8. **Multi-spec** requires manual coordination
9. **Agent-hostile** prose documentation enables hallucination

The alternative exists:

- **Zeitschrift**: Scope graph-based, typed AST generation
- **One IR**: All languages share the same intermediate representation
- **Validated references**: Every `$ref` must resolve or build fails
- **Runtime schemas**: Zod, Pydantic, serde validation at boundaries
- **Agent-native**: Structured output, no prose to hallucinate from

For teams that value correctness, maintainability, and the ability to trust
their generated clients, traditional OpenAPI generators are the wrong choice.
